# 테스트

## 1. 왜 테스트를 작성해야 하는가?

코드를 작성하는 것만으로도 바쁜데, 왜 테스트까지 작성해야 할까요? 실무에서 테스트가 필수인 이유를 알아봅시다.



### **버그 방지 - 조기 발견의 힘**

테스트 없이 개발하면 버그를 고객이 발견합니다. 테스트가 있으면 개발자가 먼저 발견합니다.

```javascript
// 실제 코드
function calculateDiscount(price, discountRate) {
  return price * discountRate;  // 🐛 버그: 할인액이 아니라 할인율을 곱함
}

// 테스트가 없다면?
// → 사용자가 10,000원 상품을 10% 할인받으려 했는데 
//   1,000원 대신 100,000원이 청구됨

// 테스트가 있다면?
test('10% 할인 계산', () => {
  expect(calculateDiscount(10000, 0.1)).toBe(1000);
});
// → 개발 단계에서 즉시 발견!
```

**실무 사례**: 한 전자상거래 회사에서 할인 로직 버그로 인해 하루 만에 수억 원의 손실이 발생했습니다. 테스트가 있었다면 배포 전에 발견할 수 있었던 문제였습니다.





### **리팩토링 안전망 - 자신감 있게 코드 개선하기**

테스트는 "안전벨트"입니다. 코드를 변경해도 기존 기능이 여전히 작동하는지 즉시 확인할 수 있습니다.

```javascript
// 기존 코드 (비효율적)
function findUser(users, id) {
  for (let i = 0; i < users.length; i++) {
    if (users[i].id === id) {
      return users[i];
    }
  }
  return null;
}

// 리팩토링 후 (개선)
function findUser(users, id) {
  return users.find(user => user.id === id) || null;
}

// 테스트가 있다면?
test('ID로 사용자 찾기', () => {
  const users = [
    { id: 1, name: 'Alice' },
    { id: 2, name: 'Bob' }
  ];
  expect(findUser(users, 2)).toEqual({ id: 2, name: 'Bob' });
});
// → 리팩토링 후에도 테스트 통과! 안전하게 변경 완료
```





### **문서화 효과 - 살아있는 사용 설명서**

잘 작성된 테스트는 코드가 "무엇을 하는지" 설명하는 문서입니다.

```javascript
describe('비밀번호 검증', () => {
  test('최소 8자 이상이어야 한다', () => {
    expect(validatePassword('abc123')).toBe(false);
    expect(validatePassword('abc12345')).toBe(true);
  });

  test('대문자를 최소 1개 포함해야 한다', () => {
    expect(validatePassword('abc12345')).toBe(false);
    expect(validatePassword('Abc12345')).toBe(true);
  });
});
// → 새로운 팀원이 이 테스트만 보면 비밀번호 규칙을 즉시 이해!
```





### **개발 속도 향상 - 장기적 관점**

초기에는 테스트 작성에 시간이 걸리지만, 장기적으로는 다음과 같은 이유로 개발 속도가 빨라집니다:

- 버그 수정 시간 감소 (운영 환경 버그는 수정 비용이 10배 이상)
- 회귀 테스트 자동화 (수동 테스트 불필요)
- 자신감 있는 배포 (금요일 오후에도 안심하고 배포 가능)

### **단위 테스트 (Unit Test)**

가장 작은 단위(함수, 메서드, 클래스)를 독립적으로 테스트합니다.

```javascript
// 단위 테스트 예시
function add(a, b) {
  return a + b;
}

test('두 수를 더한다', () => {
  expect(add(2, 3)).toBe(5);
});
```

**특징**:

- **빠름**: 밀리초 단위로 실행
- **독립적**: 다른 코드에 의존하지 않음
- **많음**: 전체 테스트의 70-80%를 차지



### **통합 테스트 (Integration Test)**

여러 모듈이 함께 작동하는지 테스트합니다.

```javascript
// 통합 테스트 예시
test('사용자 등록 시 데이터베이스에 저장되고 이메일이 발송된다', async () => {
  // UserService + Database + EmailService 모두 함께 테스트
  const user = await userService.register({
    email: 'test@test.com',
    password: '12345678'
  });
  
  expect(user).toBeDefined();
  expect(await database.findUser(user.id)).toBeTruthy();
  expect(emailService.sentEmails).toHaveLength(1);
});
```

**특징**:

- **느림**: 초 단위로 실행
- **의존적**: 여러 모듈 간 상호작용 검증
- **적음**: 전체 테스트의 20-30%



### **Mock (모의 객체)**

실제 객체를 흉내 내는 가짜 객체입니다. 외부 의존성 없이 테스트할 수 있게 해줍니다.

```javascript
// 실제 상황: API 호출이 필요함
async function getUserData(userId) {
  const response = await fetch(`https://api.example.com/users/${userId}`);
  return response.json();
}

// 테스트할 때마다 실제 API를 호출하면?
// - 네트워크 필요 (느림)
// - API 서버 필요 (비용)
// - 결과가 매번 다를 수 있음 (불안정)

// Mock을 사용하면?
test('사용자 데이터 가져오기', async () => {
  // fetch를 가짜로 대체
  global.fetch = jest.fn(() =>
    Promise.resolve({
      json: () => Promise.resolve({ id: 1, name: 'John' })
    })
  );

  const user = await getUserData(1);
  expect(user.name).toBe('John');
  // 빠르고, 안정적이고, 외부 의존성 없음!
});
```

**정리**:

- **단위 테스트**: 작고 빠른 테스트 (함수 하나)
- **통합 테스트**: 크고 느린 테스트 (여러 모듈)
- **Mock**: 외부 의존성을 가짜로 대체
